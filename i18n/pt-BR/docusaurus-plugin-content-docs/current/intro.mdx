A família de pacotes da [`@testing-library`][npm] ajuda você a testar
componentes da IU de uma forma centrada no usuário.

> [Quanto mais seus testes se assemelham à forma como o software é usado, mais confiança eles podem lhe dar.](guiding-principles.mdx)

## O Problema

Você deseja escrever testes de manutenção que lhe forneça alta confiança de que
seus componentes estão funcionando para os usuários. Como parte desse objetivo,
você deseja que os testes evitem incluir detalhes de implementação, para que as
refatorações dos seus componentes (mudanças na forma de implementar, e não de
funcionalidade) não quebrem seus testes e atrasem você e sua equipe

## A solução

O core da [`DOM Testing Library`][dom], é uma solução leve para testar paginas
web consultando e interagindo com os nós do DOM (sejam simulados com
[`JSDOM`][jsdom]/[Jest][jest] ou no navegador). Os principais utilitários
consiste em consultar o DOM em busca de nós de uma maneira semelhante como um
usuário encontra esses elementos na página. Dessa forma, a biblioteca ajuda a
garantir que seus testes dão a você a confiança de que sua aplicação irá
funcionar quando um usuário real usar ele.

O core da biblioteca foi agrupada para fornecer APIs fáceis para diversos
frameworks, incluindo [React][react], [Angular][angular] e [Vue][vue]. Também
existe um plugin para usar as buscas da testing-library para testes end-to-end
em [Cypress][cypress] e uma implementação para [React Native][react-native].

### O que essa biblioteca não é

1.  Um executor de testes ou framework
2.  Específico para um framework de teste

`DOM Testing Library` funciona com qualquer ambiente que fornece APIs do DOM,
como Jest, Mocha + JSDOM ou um navegador

### O que você deve evitar com o Testing Library

A Testing Library incentiva você a evitar testes de
[detalhes de implementação](https://kentcdodds.com/blog/testing-implementation-details)
como as partes internas de um componente que você está testando (embora ainda
seja possível). [Os Princípios Orientadores](/docs/guiding-principles) desta
biblioteca enfatiza na concentração de testes que se assemelham muito à forma
como suas páginas da web interagem pelos usuários.

Você pode querer evitar os seguintes detalhes de implementação:

1. Estado interno de um componente
1. Métodos internos de um componente
1. Métodos de ciclo de vida de um componente
1. Componentes filhos
